# design — snd-ygot-pipeline

## Context

生成管线现状（explore 实证，2026-07-09）：

- `backend/internal/generated/huawei/huawei.go`（build-tag `generate`）：go:generate 输出 `all.gen.go` + 两条 BSD sed（`sed -i ''`，Linux 报错）+ gofmt；但仓库实际是手拆 6 文件（`types.go`/`device.go`/`enums.gen.go`/`ifm.gen.go`/`system.gen.go`/`vlan.gen.go`，commit 68731a8），指令与产物脱节。
- `fix/fix_enum.go`：跨平台 Go 修复器已写好、从未接线（零调用点）。
- `backend/internal/generated/openconfig/openconfig.go`：第二条手工 go:generate（输入 `backend/internal/yang/models/`），被 `yangschema/load.go`（生产）+ `netconf.go` openconfig 遗留分支消费。
- `backend/internal/yang/generate.go`：死管线（输出目录空、零消费者）；同目录 `models/` 是 openconfig 生成的输入，**不是死的**。
- CI `compliance.yml` R04 step：无条件 fail 任何 `backend/internal/generated/` diff → 合法 regen 无法合入（门禁死锁）。
- `pr-size.yml` 排除清单只有前端生成物（`*.gen.ts` 等），不含 Go 生成物。
- **可复现性已实证**：Linux 实跑 ygot v0.34.0（go.mod 锁版）+ enum 修复 + gofmt → 与已提交 6 文件顶层声明集合完全一致（仅差手拆时加的 25 行 `var _` 占位符）、ySchema gzip 字节 md5 一致。
- yang-models 是仅构建期 submodule，worktree 内默认未初始化（`git submodule update --init yang-models` 后可用）。

约束：R04（禁手写 YANG 结构体）、R17（spec-first）、单 commit ≤500 行（手写代码）、T05/T06 测试军规、W07（禁 force push）。

## Goals / Non-Goals

**Goals:**
- 生成管线可复现：任何人在 Linux/macOS 上一条命令重出与仓库一致的生成物。
- 参数化：加厂商 = 加一份声明式生成配置（gen.conf），零脚本改动——对齐 P5-4「首个非 Huawei 驱动」的前置需求。
- R04 门禁从「冻结 generated/」进化为「regen-and-diff 漂移验证」，解开死锁且比冻结更强（防手改混入生成物）。
- 消灭 macOS-only sed、注释/README 漂移、死管线。

**Non-Goals:**
- 不改任何运行时行为（生成物语义等价，调用方零改动）。
- 不动 `internal/yang/models/` 的位置（openconfig 生成输入，搬家是无谓 churn）。
- 不做 P5-4 内容：不加新厂商、不收敛 mergeConfig/config_delete type-switch。
- 不升级 ygot 版本（锁 v0.34.0；升级是独立 change，届时门禁会强制同步 regen）。

## Decisions

### D1 manifest 形态：每厂商目录一份 `gen.conf`，脚本扫描驱动

`backend/internal/generated/<pkg>/gen.conf`（KV 格式）：

```
yang_path=yang-models/network-router/8.20.10/ne40e-x8x16   # 相对仓库根
modules=huawei-vlan huawei-ifm huawei-system huawei-pub-type huawei-extension
generate_fakeroot=true
compress_paths=false
```

`scripts/gen-yang.sh`（POSIX sh）扫描 `backend/internal/generated/*/gen.conf`，对每个包执行：ygot generator（`go run github.com/openconfig/ygot/generator`，版本由 go.mod 锁定）→ 输出固定 `all.gen.go`、package 名 = 目录名 → genfix 后处理 → gofmt。根 `Makefile` 加 `gen-yang`（全量）与 `gen-yang VENDOR=<pkg>`（单包）薄封装。

- 为什么不是 Makefile 变量 manifest：加厂商要改中枢 Makefile，非「加数据」；gen.conf 与包同目录自描述，对齐 driver.Descriptor「加模块=注册一条数据」的心智。
- 为什么不是保留 go:generate：go:generate 行是本次脱节事故的事发地——两处事实源（指令注释 vs 实际布局）必然再漂移。**事实源唯一化到 gen.conf + 脚本**，删除 `huawei.go` 与 `openconfig.go` 里的 go:generate 文件。
- submodule 前置校验：脚本开头检测 `yang_path` 目录存在且非空，缺失时报错并给出 `git submodule update --init yang-models` 指引（R08 可操作降级）。

### D2 后处理器：泛化 `fix_enum.go` 为 `backend/tools/genfix`

现 `fix/fix_enum.go` regex 绑死 `Huawei\w+`。泛化为「Go 标识符内 `|` → `_OR_`」的通用修复（只作用于 ygot 生成的枚举标识符模式），对所有厂商输出统一执行——无匹配时是幂等 no-op，不需要 per-vendor 开关。删除 `fix/` 目录与 build-tag，`genfix` 是正常 main 包（可测试）。

- 位置 `backend/tools/genfix`：对齐既有构建期 codegen 工具惯例（`backend/tools/tasknamegen`）；仓库根 `scripts/` 无 Go module，`go run` 需在 backend module 内。
- 为什么不用 sed：BSD/GNU 语法分裂（本次事故根因之一）；Go 工具跨平台且可表格驱动测试（B1）。
- **第二职责：生成头部机器路径规范化**（apply spike 实证）：ygot 输出头部含生成机器绝对路径（`This package was generated by /Users/leezesi/go/pkg/mod/.../names.go`）与 `-path` 参数原样回显——跨机器 regen 必然 diff，直接击穿 CI regen-and-diff 门禁。genfix 将该行规范化为机器无关形式（如 `generated by ygot@<go.mod 版本>`），`-path` 回显由脚本固定 cwd/canonical 路径保证确定性。

### D3 生成物布局：收敛回单 `all.gen.go`

用真实管线重出 `all.gen.go`，删除手拆 6 文件。手拆唯一动机是可读性，但代价是管线不可复现——生成物不是给人读的（R04 心智），单文件与 ygot 原生输出一致，永不再拆。`doc.go` 保留并更新（package 文档，非生成物）。

### D4 R04 门禁：regen-and-diff 漂移验证（compliance.yml 原位替换）

替换现「Check Generated Code (R04)」step：若 PR 触碰 `backend/internal/generated/**`、`scripts/gen-yang.sh`、`scripts/genfix/**`、`yang-models`（submodule 指针）之一 → `git submodule update --init yang-models` + `make gen-yang` + `git diff --exit-code backend/internal/generated/`；未触碰则跳过（省 CI 时间）。语义：**生成物改动合法当且仅当可由管线复现**——比冻结更强（冻结挡不住「管线外手改+门禁豁免合入」的历史路径，regen-diff 挡得住）。

- 同 PR 自举可行：pull_request 事件使用 PR 分支的 workflow 定义，改门禁与改生成物同 PR 生效。
- 先例：`gen-contract` 漂移门禁（contract-drift.yml）。
- **本地 L3 层对称（apply 期实证补充）**：`.githooks/pre-commit` 的 R04 段是同一冻结死锁的本地镜像（无条件拒绝 generated/ 暂存），同步改为「触及生成物/manifest 时本地 regen + diff 校验」（README/doc.go 纯文档不触发）；另修复其 R15 段对包退役提交（目录不再含 Go 文件）的 no-packages 误判——这两处不修，本 change 自身无法提交（门禁死锁的活体证据）。

### D5 pr-size 排除后端生成物

排除清单追加 `:(exclude)backend/internal/generated/**`（**整目录**，非仅 all.gen.go）：待删除的手拆遗留文件（types.go/device.go）不带 `.gen.go` 后缀，按模式排除会漏计 4900+ 行删除；该目录由 R04 regen-and-diff 门禁守护，手写内容仅 gen.conf/README/doc.go（体量可忽略）。先例 #60 前端 `*.gen.ts`。

### D6 提交切分（单 PR，原子 commit 序列）

1. 门禁先行：pr-size 排除 + compliance R04 step 重写（regen-and-diff）。
2. 管线：`scripts/gen-yang.sh` + `scripts/genfix/`（含 B1 测试）+ gen.conf×2 + Makefile target + 删 go:generate 文件/死管线 + README/doc.go 修正。
3. regen：huawei 6 文件 → all.gen.go；openconfig regen（若有 diff，见 R3）。

手写代码合计约 300–400 行，生成物 churn 被 pr-size 排除；每 commit What/Why/How 三段式。

## Risks / Trade-offs

- **[R1] genfix 泛化 regex 误伤其他生成代码** → B1 表格驱动测试（`|` 替换正常/FlexE 变体/无匹配幂等/非枚举行不动）；且 huawei regen diff 必须为零（等价性由既有 xmlcodec golden 全量对拍 + B2 集成 + 全量单测三重兜底）。
- **[R2] CI regen job 耗时**（go run 首次编译 ygot generator 约 1–2 min）→ 条件触发（仅相关路径变更时跑）+ 既有 Go module cache。
- **[R3] openconfig 包 regen 可复现性未 spike** → **已解答（apply 任务 2.1，2026-07-09）**：regen 与已提交 all.gen.go 字节级一致，仅 2 行头部注释差异（生成机器路径 + `-path` 回显格式）——openconfig 直接纳入管线与门禁；头部差异由 D2 规范化消除。
- **[R4] 未来 ygot 升级产生全量 regen diff** → 这是门禁的预期行为（强制生成物与管线同步），非缺陷；升级走独立 change。
- **[R5] 开发者本机未初始化 submodule** → 脚本前置校验 + 可操作报错指引；`make setup`（TM08）已含 submodule 初始化则天然覆盖。
- **权衡：删除 go:generate 惯例** — 失去 `go generate ./...` 标准入口，换取事实源唯一；`doc.go` 与 README 明确指向 `make gen-yang`。

## Migration Plan

同 PR 内「门禁 → 管线 → regen」三段推进（D6），合入即完成，无部署/回滚问题（纯构建期 + CI 改动，运行时零变更）。回退 = revert PR。
