## 一、激活时机（何时自动触发）
1.  当用户需求包含「Commit」「提交日志」「版本更新」等关键词时，自动激活。
2.  开发流程中，代码评审通过后，自动触发本技能，生成标准三段式 Commit 信息。
3.  每次提交仅对应一个原子功能/BUG修复，自动校验代码行数，不符合则拒绝生成 Commit。

## 二、核心原则（底层设计逻辑）
1.  清晰可追溯原则：Commit 信息需明确说明“做了什么、为什么做、怎么做”，便于后续追溯、排查问题。
2.  小步迭代原则：单次 Commit 仅对应一个小功能/BUG，避免多功能合并，降低代码回滚风险。
3.  语义统一原则：格式固定，语言简洁，不冗余、不模糊，团队可快速理解变更内容。
4.  合规性原则：与项目全局规则联动，自动校验代码行数，不符合则禁止提交。

### 样例2：技能输出效果（标准三段式）
What
新增 Go TTL+LRU 高性能内存缓存模块，实现 Set、Get、Invalidate、定时清理过期缓存功能，Key 为设备 IP+YANG 节点路径
Why
项目无数据库，需通过内存缓存减少 NETCONF 请求频率，提升接口响应性能；同时需保证缓存时效性，避免脏数据，因此实现 TTL 自动老化 + LRU 淘汰机制
How
使用 sync.RWMutex 保证协程安全，避免并发读写竞态；
采用 map 存储缓存项，包含 value 和 expireTime，实现 TTL 自动过期；
维护 LRU 队列，超过最大容量时淘汰最久未使用的缓存项；
启动定时协程（每 10 秒）清理过期缓存，确保内存不溢出；
提供 Invalidate 方法，支持配置下发后主动失效对应缓存；
代码共 386 行，符合单次代码 < 500 行约束。

### 样例3：BUG修复场景（输出效果）
What
修复 NETCONF 客户端并发连接时的数据竞态问题，解决多协程调用 Connect () 导致的 driver 重复赋值异常
Why
多协程并发调用 NETCONF 客户端的 Connect () 方法时，未加锁保护，导致 driver 被重复赋值，引发程序 panic，影响设备连接稳定性，需紧急修复
How
在 NETCONFClient 结构体中添加 sync.Mutex 锁；
在 Connect () 方法中添加锁的加锁 / 解锁操作，确保并发安全；
增加连接状态判断，避免重复发起连接请求；
补充协程 panic 防护，添加 defer recover ()，防止协程崩溃；
代码修改量 86 行，符合单次代码 < 500 行约束，不影响其他功能。
