## 一、激活时机（何时自动触发）
1.  当用户需求包含「Commit」「提交日志」「版本更新」等关键词时，自动激活。
2.  开发流程中，代码评审通过后，自动触发本技能，生成标准提交信息。
3.  每次提交仅对应一个原子功能/BUG修复，自动校验代码行数，不符合则拒绝生成。

## 二、核心原则（底层设计逻辑）
1.  **约定式提交前缀**：所有提交**必须**以 `<type>: ` 开头，清晰标识变更类型。
2.  **中文统一原则**：所有提交信息**必须全程使用中文**，保持项目风格统一。
3.  清晰可追溯原则：提交信息需明确说明"做了什么、为什么做、怎么做"。
4.  小步迭代原则：单次提交仅对应一个原子变更，避免多功能合并。
5.  合规性原则：自动校验代码行数，单次代码 < 500 行。

## 三、Type 前缀规范

| 前缀 | 适用场景 |
|------|---------|
| `feat:` | 新增功能、新特性 |
| `fix:` | Bug 修复 |
| `docs:` | 文档变更（README、设计文档、技能定义等） |
| `test:` | 测试新增、测试修复、测试重构 |
| `refactor:` | 代码重构（不影响功能、非 Bug 修复） |
| `style:` | 代码格式化、样式调整（不影响代码逻辑） |
| `chore:` | 构建工具、依赖升级、CI 配置等 |
| `perf:` | 性能优化 |

## 四、完整格式规范

```
<type>: <subject>

What: <修改内容概要，一句话说清楚改了什么>
Why: <为什么要改，解决什么问题，背景和动机>
How: <具体实现方式，技术方案说明>
```

**强制规则：**
- Subject 不超过 50 字符，中文，结尾不加句号
- What/Why/How 每段 1-3 句话，简洁明了
- 关键文件、关键改动在 How 中明确列出

## 五、使用样例（实操指令+输出效果）

### 样例1：新功能开发
```
feat: 实现 YANG 模型驱动动态表单渲染框架

What: 新增 YangRenderer/YangPanel/YangTable/YangField 组件体系，支持通过 Schema 定义零代码生成配置页面。
Why: 每个 YANG 模块都手写表单重复劳动量大，新增模块效率低。统一渲染框架可大幅提升开发效率，保证交互一致性。
How: 创建 yang-schema.ts 类型系统定义 Schema 节点；实现 YangRenderer 统一调度器，根据节点类型分发渲染；开发 YangTable 支持列表数据的 CRUD 操作；代码共 860 行，分 3 次提交。
```

### 样例2：Bug 修复
```
fix: 修复 CORS 端口不匹配导致前端设备树为空

What: 在后端测试服务器的 CORS 配置中添加 3000 端口支持，同步修正测试框架端口配置。
Why: 手动验收时发现前端设备树为空，但 E2E 测试全部通过，根因是 CORS 只配置了 5173 端口，而 Vite 实际运行在 3000。
How: 在 cmd/test-server/main.go 的 CORS AllowOrigins 数组中添加 3000 端口；修正 playwright.config.ts baseURL 与实际端口一致。
```

### 样例3：测试相关
```
test: 新增 VLAN 创建流程 E2E 测试用例

What: 新增 5 个 VLAN 管理端到端测试用例，覆盖页面加载、数据展示、新建、编辑、删除全流程。
Why: VLAN 功能是核心配置模块，缺少自动化测试容易出现回归问题；重构动态表单前需要完整测试保障。
How: 在 web/tests/vlan.spec.ts 中新增完整测试套件；使用 Playwright Page Object 模式封装操作；测试依赖后端测试服务器 + NETCONF 模拟器。
```

### 样例4：文档更新
```
docs: 新增 E2E 测试避坑指南到 TESTING.md

What: 在前端测试指南中新增"E2E 测试避坑指南"章节，记录本次 CORS 端口不匹配问题的完整案例和解决方案。
Why: 本次"测试通过但验收失败"的问题具有典型性，需要固化为经验，避免后续踩同样的坑。
How: 总结 3 条 E2E 测试黄金法则；添加错误写法 vs 正确写法代码对比；在开发流程最前端添加"环境一致性检查"强制关卡。
```
