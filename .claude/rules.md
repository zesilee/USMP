# 项目全局不可违反铁则（永久生效）
## 一、开发模式铁则（优先级最高）
1.  全程强制使用 Claude Code Plan 模式迭代，所有功能先拆分原子任务Plan，再启动开发，禁止无规划开发。
2.  严格遵循 TDD 测试驱动开发：**必须先编写测试用例，测试用例确认后，再编写业务实现代码**，禁止先写代码后补测试。
3.  单次输出代码量 **严格小于 500 行**，禁止一次性堆砌大段代码，小步迭代、频繁提交。
4.  所有代码编写完成后，必须执行完整 Code Review，Review 不通过，绝不允许提交代码。
5.  禁止跳过任何开发流程（Plan → 测试 → 代码 → Review → Commit），禁止合并流程、省略步骤。

## 二、架构约束铁则（不可变更）
1.  后端语言与框架：固定使用 Go 1.21+，基于 protoactor-go 实现 Actor 模型，禁止更换框架。
2.  设备管控架构：**一台交换机对应一个独立 DeviceActor 实例**，由 ManagerActor 统一管控（注册、路由、销毁），禁止多设备共用一个Actor。
3.  YANG 模型处理：固定使用 ygot 工具，根据 YANG 描述文件自动生成 Go 强类型结构体，禁止手写 YANG 相关结构体。
4.  数据存储约束：**无任何数据库**（禁止使用 MySQL、SQLite、Redis 等任何存储组件），仅允许：
    - 内存缓存（带 TTL 老化 + LRU 淘汰）；
    - 本地 JSON 文件（仅存储设备连接元信息，不存储设备运行配置）。
5.  设备配置约束：所有设备配置**仅在运行时通过 NETCONF 协议实时从交换机读取**，不持久化落盘，缓存过期自动重新拉取。
6.  通信协议：固定使用 NETCONF 协议（RFC 6241 标准），SSH 830 端口，禁止使用 Telnet、SNMP 等其他协议对接交换机。
7.  前端约束：根据 YANG 模型自动渲染动态表单控件，禁止手写固定配置页面。

## 三、技术实现铁则
1.  内存缓存：必须实现 TTL 自动老化 + LRU 淘汰机制，协程安全，Key 固定为「设备IP+YANG节点路径」，配置下发后主动失效对应缓存。
2.  Actor 规范：严格遵循 protoactor-go 生命周期管理，消息异步处理，禁止阻塞长调用，处理超时、离线、异常熔断场景，避免协程泄漏。
3.  ygot 规范：严格遵循 YANG 树形层级（container/leaf/list/enum），实现 ygot 对象与 NETCONF XML 双向无损转换，禁止使用 interface{} 泛型。
4.  NETCONF 规范：支持 get-config、edit-config、commit、lock/unlock 操作，实现断线自动重连、超时重试、异常错误处理。

## 四、提交规范铁则
1.  每次 Git Commit 必须严格遵循「What/Why/How」三段式格式，禁止简化、省略任何一段。
2.  What：明确本次变更的具体功能点、BUG 修复内容，不模糊、不冗余。
3.  Why：说明变更的业务背景、解决的痛点、架构必要性，禁止无理由提交。
4.  How：简要说明技术实现逻辑、改动范围、核心交互流程，贴合本次 ≤500 行代码变更。
5.  单次 Commit 仅对应一个原子功能/BUG 修复，禁止合并多个不相关功能一次性提交。

## 五、通用约束铁则
1.  禁止引入无关第三方依赖，仅保留项目必需依赖（protoactor-go、ygot、Gin、NETCONF 相关库）。
2.  所有代码必须保证协程安全、无数据竞态、无内存泄漏，做好 panic 防护。
3.  严格遵循 Go 代码规范，命名统一、注释清晰，禁止冗余代码、过度封装。
4.  所有异常场景（设备离线、NETCONF 连接失败、缓存过期、Actor 异常）必须有降级处理，禁止程序崩溃。
