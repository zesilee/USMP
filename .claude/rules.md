# 项目全局不可违反铁则（永久生效）
## 一、开发模式铁则（优先级最高）
1.  全程强制使用 Claude Code Plan 模式迭代，所有功能先拆分原子任务Plan，再启动开发，禁止无规划开发。
2.  严格遵循 TDD 测试驱动开发：**必须先编写测试用例，测试用例确认后，再编写业务实现代码**，禁止先写代码后补测试。
3.  **硬性约束：单次输出代码量严格 ≤ 500 行**，禁止一次性堆砌大段代码。**必须**小步迭代，**每次迭代完成一个完整原子功能且测试通过后必须立即提交代码**，禁止积累多个功能再提交。
4.  所有代码编写完成后，必须执行完整 Code Review，Review 不通过，绝不允许提交代码。
5.  禁止跳过任何开发流程（Plan → 测试 → 代码 → Review → Commit），禁止合并流程、省略步骤。

## 二、架构约束铁则（不可变更）
1.  后端语言与框架：固定使用 Go 1.21+，基于 **yang-controller-runtime**（controller-runtime 风格架构），框架处理所有 boilerplate，禁止更换架构。
2.  设备管控架构：**connection pool 复用设备连接**，每个 YANG 模块对应一个 Controller，由 Manager 统一管控，禁止使用旧 Actor 模型。
3.  YANG 模型处理：固定使用 ygot 工具，根据 YANG 描述文件自动生成 Go 强类型结构体，禁止手写 YANG 相关结构体。
4.  数据存储约束：**无任何数据库**（禁止使用 MySQL、SQLite、Redis 等任何存储组件），仅允许：
    - 内存缓存（带 TTL 老化 + LRU 淘汰）；
    - 本地 JSON 文件（仅存储设备连接元信息，不存储设备运行配置）。
5.  设备配置约束：所有设备配置**仅在运行时通过协议实时从交换机读取**，不持久化落盘，缓存过期自动重新拉取。
6.  通信协议：支持 **NETCONF（RFC 6241 标准，SSH 830 端口） + gNMI（openconfig/gnmi，默认 9339/9340 端口）** 双协议，禁止使用 Telnet、SNMP 等其他协议对接交换机。
7.  前端约束：
    - 配置页面：根据 YANG 模型自动渲染动态表单控件，禁止手写固定配置页面。
    - UI/UX 设计：**涉及视觉设计、页面美化、交互优化，自动触发 `web-design-engineer` skill**，遵循设计系统规范。
    - 数据可视化：图表、仪表盘等视觉展示，自动触发 `web-design-engineer` skill。
    - 禁止纯后端逻辑、CLI 工具、数据处理脚本等非视觉任务使用设计 skill。

## 三、前端技能触发规则（自动生效）
1.  **触发 `web-design-engineer` skill 的场景**（满足任一自动触发）
    - 页面美化、UI/UX 优化、响应式设计
    - 登录页、仪表盘、卡片布局、图表展示
    - 动画、交互原型、数据可视化
    - 任何涉及「视觉效果、设计风格、用户体验」的前端开发
2.  **不触发的场景**
    - 纯逻辑开发：状态管理、API 对接、组件封装
    - 工程化：构建配置、测试框架、性能优化
    - 纯功能：表单校验、路由配置、权限控制
3.  **设计流程约束**
    - 必须先声明设计系统（色彩、排版、间距、阴影、动效），经确认后再写代码
    - 尽早出 v0 草案：核心结构 + 色彩/排版 token + 关键模块占位符
    - 提供至少 2 个设计变体供选择，保守 + 激进组合
    - 禁止：AI 风格陈词滥调（紫粉蓝渐变、左边框圆角卡片、滥用 Inter/Roboto 字体、无意义数字/图标）
    - 无真实图标/图片时，使用规范占位符，禁止用 emoji 替代图标

## 四、技术实现铁则
1.  内存缓存：必须实现 TTL 自动老化 + LRU 淘汰机制，协程安全，Key 固定为「设备IP+YANG节点路径」，配置下发后主动失效对应缓存。
2.  yang-controller-runtime 规范：严格遵循 Manager → Controller → Reconciler → Source 架构，Reconciler 保持无状态，框架处理并发排队和限频。
3.  ygot 规范：严格遵循 YANG 树形层级（container/leaf/list/enum），实现 ygot 对象与协议消息双向转换，禁止滥用 interface{} 泛型。
4.  协议规范：支持 Get/Set/Subscribe 操作，连接池实现断线自动重连、超时重试、异常错误处理。

## 五、提交规范铁则
1.  **每次迭代完成一个完整原子功能且测试通过后必须提交**，禁止积累多个功能。
2.  每次 Git Commit 必须严格遵循「What/Why/How」三段式格式，禁止简化、省略任何一段。
3.  What：明确本次变更的具体功能点、BUG 修复内容，不模糊、不冗余。
4.  Why：说明变更的业务背景、解决的痛点、架构必要性，禁止无理由提交。
5.  How：简要说明技术实现逻辑、改动范围、核心交互流程，贴合本次 ≤500 行代码变更。
6.  单次 Commit 仅对应一个原子功能/BUG 修复，禁止合并多个不相关功能一次性提交。

## 六、通用约束铁则
1.  禁止引入无关第三方依赖，仅保留项目必需依赖。
2.  所有代码必须保证协程安全、无数据竞态、无内存泄漏，做好 panic 防护。
3.  严格遵循 Go 代码规范，命名统一、注释清晰，禁止冗余代码、过度封装。
4.  所有异常场景（设备离线、连接失败、缓存过期）必须有降级处理，禁止程序崩溃。
